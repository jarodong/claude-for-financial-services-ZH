## 模板要求

**此 Skill 使用模板构建 LBO 模型。始终首先检查是否有附件模板文件。**

开始任何 LBO 模型之前：
1. **如果有附件/提供的模板**：精确使用该模板结构——复制并填入用户数据
2. **如果没有附件模板**：询问用户：*"您有特定的 LBO 模板希望我使用吗？如果没有，我可以使用标准模板，包含资金来源与用途、运营模型、债务计划和回报分析。"*
3. **如果使用标准模板**：复制 `examples/LBO_Model.xlsx` 作为起点，填入用户假设

**重要**：当附件为 `LBO_Model.xlsx` 等文件时，必须将其用作模板——不要从头构建。即使模板看起来复杂或功能超出需要，也要复制并适应用户需求。有模板时绝不决定"从头构建"。

## Claude 关键指令——先读此处

### 环境：Office JS vs Python

**如果在 Excel 中运行（Office 加载项 / Office JS 环境）：**
- 直接使用 Office JS（`Excel.run(async (context) => {...})`）——不要使用 Python/openpyxl
- 通过 `range.formulas = [["=B5*B6"]]` 写入公式——Office JS 公式在活动工作簿中原生重算
- 同样的公式优先规则适用：对计算单元格设置 `range.formulas`，绝不使用 `range.values`
- 使用 `range.format.font.color` / `range.format.fill.color` 设置蓝/黑/紫/绿约定
- 无需单独重算步骤——Excel 原生处理计算
- **合并单元格陷阱：** 不要先调用 `.merge()` 再在合并范围上设置 `.values`（会抛出 `InvalidArgument`——范围仍报告原始尺寸）。改为：先在左上角单元格写入值（`ws.getRange("A7").values = [["SOURCES & USES"]]`），然后合并+格式化整个范围（`ws.getRange("A7:F7").merge(); ws.getRange("A7:F7").format.fill.color = "#1F4E79";`）

**如果生成独立 .xlsx 文件（无活动 Excel 会话）：**
- 使用 Python/openpyxl
- 写入公式字符串（`ws["D20"] = "=B5*B6"`），然后在交付前运行 `recalc.py`

此 Skill 其余部分使用 openpyxl 示例，但相同原则适用于 Office JS——只需转换 API 调用。

### 核心原则
* **每个计算必须是 Excel 公式**——绝不从 Python 计算值并硬编码结果到单元格。使用 openpyxl 时，写 `cell.value = "=B5*B6"`（公式字符串），而非 `cell.value = 1250`（计算结果）。模型必须是动态的，输入变化时自动更新。
* **使用模板结构**——遵循 `examples/LBO_Model.xlsx` 或用户提供的模板中的组织方式。不要自创布局。
* **使用正确的单元格引用**——所有公式应引用适当单元格。绝不输入应来自其他单元格的数字。
* **保持符号约定一致性**——遵循模板使用的符号约定（有些流出用负数，有些用正数）。全程一致。
* **逐节工作，每步与用户验证**——完整完成一节，向用户展示构建内容，运行该节验证检查，获得确认后再进入下一节。不要端到端构建整个模型再展示——后续节依赖前面的节，在回报分析建好后发现资金来源与用途的错误意味着到处返工。

### 公式颜色约定
* **蓝色（0000FF）**：硬编码输入——手动输入的数字，不引用其他单元格
* **黑色（000000）**：含计算的公式——使用运算符或函数的任何公式（`=B4*B5`、`=SUM()`、`=-MAX(0,B4)`）
* **紫色（800080）**：链接到**同一工作表**的单元格——无计算的直接引用（`=B9`、`=B45`）
* **绿色（008000）**：链接到**不同工作表**的单元格——跨表引用（`=Assumptions!B5`、`='Operating Model'!C10`）

### 填充色板——专业蓝灰色调（默认，除非用户/模板另行指定）
* **保持极简**——仅使用蓝色和灰色作为单元格填充。不要引入绿色、黄色、红色或多种强调色。专业 LBO 模型讲究克制。
* **默认填充色板：**
  * **节标题**（资金来源与用途、运营模型等）：深蓝 `#1F4E79` 配白色粗体文本
  * **列标题**（第 1 年、第 2 年等）：浅蓝 `#D9E1F2` 配黑色粗体文本
  * **输入单元格**：浅灰 `#F2F2F2`（或白色）——蓝色*字体*是信号，填充是次要的
  * **公式/计算单元格**：白色，无填充
  * **关键输出**（IRR、MOIC、退出权益）：中蓝 `#BDD7EE` 配黑色粗体文本
* **这就是全部色板。** 3 种蓝色 + 1 种灰色 + 白色。如果模板使用自己的颜色，遵循模板。
* 注意：上述蓝/黑/紫/绿**字体**颜色用于区分输入 vs 公式 vs 链接。这与**填充**色板是分开的——两者配合使用。

### 数字格式标准
* **货币**：`¥#,##0;(¥#,##0);"-"` 或 `¥#,##0.0`（取决于模板）
* **百分比**：`0.0%`（一位小数）
* **倍数**：`0.0"x"`（一位小数）
* **MOIC/详细比率**：`0.00"x"`（两位小数，精确）
* **所有数字单元格**：右对齐

## 模板分析阶段——先做此事

填充公式之前，彻底检查模板：

1. **绘制结构图**——识别每节位置及其相互关系。注意哪些节馈入其他节。

2. **理解时间线**——哪些列代表哪些期间？有"交割"或"模拟备考"列吗？预测期从哪里开始？

3. **识别输入 vs 公式单元格**——模板通常使用颜色编码、边框或阴影指示哪些单元格需要输入 vs 公式。尊重这些约定。

4. **仔细阅读现有标签**——行标签精确告诉你需要什么计算。不要假设——阅读模板要求的内容。

5. **检查现有公式**——有些模板部分填充。除非明确要求，不要覆盖有效公式。

6. **注意模板特有约定**——符号约定、小计结构、节组织方式、不同组件是否有单独工作表等。

## 填充公式——通用方法

对每个需要公式的单元格，遵循此层级：

### 第 1 步：检查模板
* 单元格已有公式吗？如果有，验证其正确性后继续。
* 有注释或说明指示预期计算吗？
* 行/列标签使计算显而易见吗？
* 相邻单元格显示应遵循的模式吗？

### 第 2 步：检查用户指示
* 用户指定了特定计算方法吗？
* 有影响此公式的既定假设吗？
* 提到过特殊要求吗？

### 第 3 步：应用标准实践
* 如果模板和用户都未指定，使用标准 LBO 建模约定
* 记录你做出的任何假设
* 如确实不确定，询问用户

## 常见问题区域

以下计算模式在 LBO 模型中经常出错。遇到时要特别注意：

### 平衡节
* 当两节必须相等时（如资金来源 = 资金用途），通常有一个"轧差项"（平衡数）
* 识别哪个项目是轧差项，将其计算为差额

### 税务计算
* 税务公式应仅引用相关收入行和税率
* 不应引用无关节（如债务计划）
* 考虑亏损是否产生税盾还是简单忽略

### 利息和循环引用
* 如果利息计算引用受现金流影响的余额，会产生循环
* 使用**期初余额**（而非平均或期末）来打破循环引用
* 模式：利息 → 现金流 → 偿还 → 期末余额（如果利息使用期末余额，会循环回来）

### 债务偿还 / 现金清偿
* 当存在多个债务层级时，通常有优先顺序
* 现金清偿应尊重优先级瀑布
* 余额不能为负——适当使用 MAX 或 MIN 函数

### 回报计算（IRR/MOIC）
* 现金流必须有正确符号：投资 = 负，收益 = 正
* 如果使用 XIRR，需要对应日期
* 如果使用 IRR，现金流应在连续期间
* MOIC = 总收益 / 总投资

### 敏感性分析表
* **使用奇数维度**（5×5 或 7×7）——绝不 4×4 或 6×6。奇数维度保证有真正的中心单元格。
* **中心单元格 = 基准情景。** 围绕模型实际假设对称构建行列轴值（如基准进入倍数 = 10.0x，轴 = `[8.0x, 9.0x, 10.0x, 11.0x, 12.0x]`）。中心单元格的 IRR/MOIC 必须等于模型实际 IRR/MOIC 输出——这证明表格连接正确。
* **高亮中心单元格**——中蓝填充（`#BDD7EE`）+ 粗体字，使基准情景视觉锚定。
* Excel 的 DATA TABLE 函数可能无法与 openpyxl 配合——改为写入引用行列标题的显式公式
* 每个单元格应显示不同值——如果全部相同，说明公式未正确变化
* 使用混合引用（如 `$A5` 行输入，`B$4` 列输入）

python /mnt/skills/public/xlsx/recalc.py model.xlsx

### 节平衡
- [ ] 必须平衡的节（资金来源/用途、资产/负债）精确平衡
- [ ] 轧差项正确计算为平衡数
- [ ] 应跨节匹配的金额一致

### 收入/运营预测
- [ ] 收入/顶线从驱动因素或增长率正确构建
- [ ] 所有成本和费用项目适当计算
- [ ] 小计和合计正确求和
- [ ] 利润率和比率合理
- [ ] 链接到假设正确

### 资产负债表（如适用）
- [ ] 资产 = 负债 + 权益（必须平衡）
- [ ] 所有项目链接到适当计划或滚动结转
- [ ] 期初余额 = 上期期末余额
- [ ] 包含检查行且显示零

### 现金流量表（如适用）
- [ ] 从正确收入数字开始
- [ ] 非现金项目适当加减
- [ ] 营运资本变化有正确符号
- [ ] 期末现金 = 期初现金 + 净现金流
- [ ] 现金余额跨报表一致

### 辅助计划
- [ ] 滚动结转计划平衡（期初 + 变化 = 期末）
- [ ] 计划正确链接到主报表
- [ ] 计算项目使用适当驱动因素
- [ ] 所有期间计算一致

### 债务/融资计划（如适用）
- [ ] 期初余额链接到资金来源或上期
- [ ] 利息按适当余额计算（通常为期初）
- [ ] 偿还尊重现金可用性和优先级
- [ ] 期末余额不能为负
- [ ] 合计正确求和各层级

### 回报/输出分析
- [ ] 退出/终值正确计算
- [ ] 包含所有相关调整
- [ ] 现金流符号正确（投资为负，收益为正）
- [ ] IRR/MOIC 公式引用完整范围
- [ ] 结果对情景而言合理

### 敏感性分析表（如适用）
- [ ] 网格维度为奇数（5×5 或 7×7）——有真正的中心单元格
- [ ] 行列轴值围绕基准情景对称（`[base-2Δ, base-Δ, base, base+Δ, base+2Δ]`）
- [ ] 中心单元格输出等于模型实际 IRR/MOIC——确认表格连接正确
- [ ] 中心单元格高亮（中蓝填充 `#BDD7EE`，粗体字）
- [ ] 行列标题包含适当输入值
- [ ] 每个数据单元格包含公式（非硬编码）
- [ ] 每个数据单元格显示不同值
- [ ] 值朝预期方向移动（更高退出倍数 → 更高 IRR 等）

### 格式
- [ ] 硬编码输入为蓝色（0000FF）
- [ ] 计算公式为黑色（000000）
- [ ] 同表链接为紫色（800080）
- [ ] 跨表链接为绿色（008000）
- [ ] 所有数字右对齐
- [ ] 全程应用适当数字格式
- [ ] 无单元格显示错误值（#REF!、#DIV/0!、#VALUE!、#NAME?）

### 逻辑合理性检查
- [ ] 数字数量级合理
- [ ] 趋势有意义（增长、下降、稳定符合预期）
- [ ] 无明显错误值（应为正数处为负、不可能的百分比等）
- [ ] 关键输出在分析类型的合理范围内

## 常见错误及避免

| 错误 | 问题 | 修复方法 |
|------|------|----------|
| 硬编码计算值 | 输入变化时模型不更新 | 始终使用引用源单元格的公式 |
| 复制后单元格引用错误 | 公式指向错误单元格 | 验证所有链接，使用适当 $ 锚定 |
| 循环引用错误 | 模型无法计算 | 对利息类计算使用期初余额，打破循环 |
| 节不平衡 | 应匹配的合计不匹配 | 确保一个项目是轧差项（计算为差额） |
| 不可能的负余额 | 支付/使用超过可用 | 适当使用 MAX(0, ...) 或 MIN 函数 |
| IRR/回报错误 | 符号错误或范围不完整 | 检查现金流符号，确保公式覆盖所有期间 |
| 敏感性表显示相同值 | 公式未随输入变化 | 检查单元格引用——需要混合引用（$A5, B$4） |
| 滚动结转不衔接 | 期初 ≠ 上期期末 | 验证期间间链接 |
| 符号约定不一致 | 加法变减法或反之 | 全程一致遵循模板约定 |

## 与用户协作——逐节检查点

* **如果模板结构不清楚**，先问再做
* **如果用户需求与模板冲突**，确认其偏好
* **完成每个主要节后**，停下来与用户验证后再继续：
  - **资金来源与用途后** → 展示平衡表，确认轧差项正确，获得签字后再建运营模型
  - **运营模型/预测后** → 展示预测损益表，确认增长率和利润率合理，获得签字后再建债务计划
  - **债务计划后** → 展示期初/期末余额和利息，确认瀑布逻辑，获得签字后再建回报分析
  - **回报分析（IRR/MOIC）后** → 展示现金流序列和输出，确认符号和范围，获得签字后再建敏感性表
  - **敏感性表后** → 展示每个单元格变化，确认基准情景落在预期位置
* **验证中发现错误**，在进入下一节前修复
* **展示工作过程**——在有帮助时解释关键公式或假设
* **绝不展示未经逐节检查的完成模型**——在源头发现错误单元格引用比从坏掉的 IRR 反向追踪更快
